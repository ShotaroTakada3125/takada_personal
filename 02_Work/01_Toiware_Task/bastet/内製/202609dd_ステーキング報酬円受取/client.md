1. DBテーブル設計
新規: staking_rewards_conversions
保持銘柄→変換先銘柄のマッピングマスタ。「受取(暗号資産)」の選択肢も同じ仕組みで表現するため、保持銘柄自身への自己参照行（例: ETH→ETH）を必ず1件登録する設計にしています。これにより「銘柄ごとにcurrencyで検索するだけで選択肢リストが揃う」というシンプルなクエリになり、Service層に特殊分岐が要りません。


CREATE TABLE staking_rewards_conversions (
    id                  BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    currency            VARCHAR(10) NOT NULL,   -- 保持銘柄
    converted_currency  VARCHAR(10) NOT NULL,   -- 変換先銘柄（保持銘柄と同一="暗号資産のまま受取"も含む）
    sort_order          INTEGER NOT NULL,       -- プルダウン表示順（0=暗号資産のまま受取 を推奨）
    delete_flag         BOOLEAN NOT NULL DEFAULT FALSE,
    register_datetime   TIMESTAMP NOT NULL,
    register_user       VARCHAR(20) NOT NULL,
    update_datetime     TIMESTAMP NOT NULL,
    update_user         VARCHAR(20) NOT NULL,
    version             INTEGER NOT NULL DEFAULT 0
);

CREATE UNIQUE INDEX ux_staking_rewards_conversions
    ON staking_rewards_conversions (currency, converted_currency);

変更: staking_rewards_entry（カラム追加）

ALTER TABLE staking_rewards_entry
    ADD COLUMN received_currency VARCHAR(10) NULL;
既存カラムis_reward_rejectionはそのまま使います（ご提示の表では「受け取りフラグ」となっていましたが、実体は"拒否フラグ"のままなのでリネーム不要、混乱回避のため名称もそのまま維持を推奨します）。
既存行はreceived_currency = NULLのままでOK（未リリース機能なのでバックフィル不要、確認済み）。
新規書き込み時の規約: isRewardRejection=falseの場合、receivedCurrencyには必ず具体的な銘柄コード（同一銘柄で受け取る場合も自分自身のコード、例"ETH"）を書き込みます。NULLは「機能導入前からの既存データ」を示すレガシー値としてのみ存在し、以後の書き込みでは発生させません。読み取り時（GET）はサーバー側でNULL → currency自身に正規化して返すため、フロントはNULLを特別扱いする必要がありません。

2. Entity/Repository（新規追加分のみ）

@Entity
@Getter @Setter
@Table(name = "staking_rewards_conversions")
public class StakingRewardsConversion extends EntityBase {
  @Id
  @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;

  @Column(nullable = false, length = 10)
  private String currency;

  @Column(nullable = false, length = 10)
  private String convertedCurrency;

  @Column(nullable = false)
  private Integer sortOrder;
}

public interface StakingRewardsConversionRepository
    extends JpaRepository<StakingRewardsConversion, Long> {

  Optional<List<StakingRewardsConversion>>
      findByCurrencyAndDeleteFlagFalseOrderBySortOrderAsc(String currency);

  int countByCurrencyAndConvertedCurrencyAndDeleteFlagFalse(String currency, String convertedCurrency);
}
StakingRewardsEntryにはreceivedCurrencyフィールド（String, nullable）を追加するだけです。

3. API形式の提案
GET /staking/{accountId}

[
  {
    "currency": "ETH",
    "conversions": ["ETH", "JPYSC", "USDC"],
    "isRewardRejection": false,
    "receivedCurrency": "JPYSC"
  },
  {
    "currency": "FLR",
    "conversions": ["FLR", "JPYSC"],
    "isRewardRejection": true,
    "receivedCurrency": null
  }
]
entry: 'denied'のような文字列1つに全状態を詰め込む方式ではなく、**isRewardRejection(boolean) + receivedCurrency(拒否時はnull)**の組み合わせにしました。型安全で、「拒否時は受取銘柄が無意味な値」であることが構造上明確になります。
フロントでのプルダウン選択値は isRewardRejection ? "DENIED" : receivedCurrency として1行で導出できるので、実装コストはほぼ同じです。
conversions配列は自己参照行込みでそのまま「拒否 + このリスト」がプルダウンの全選択肢になります。
POST /staking/update

{
  "accountId": "10000008",
  "changeList": [
    { "currency": "ETH", "isRewardRejection": false, "receivedCurrency": "JPYSC" },
    { "currency": "FLR", "isRewardRejection": true,  "receivedCurrency": null }
  ]
}
バリデーションルール:

項目	ルール
currency	staking_rewards_currencyに存在しdeleteFlag=falseであること
isRewardRejection	必須(boolean)
receivedCurrency	isRewardRejection=falseの場合必須。staking_rewards_conversionsに(currency, receivedCurrency)の組がdeleteFlag=falseで存在すること。isRewardRejection=trueの場合は値を無視しサーバー側でnull格納
旧countByCurrencyAndDeleteFlagFalseとcountByCurrencyAndJpyscConversionEnabledTrueAndDeleteFlagFalseという2つの別バリデーションが、countByCurrencyAndConvertedCurrencyAndDeleteFlagFalse1つに統合されます。
