
# Phase 1：ロジックの詳細設計（仮案）

## 1. Step A: 論理分解エンジン（Decomposition API）

このステップの目的は、ユーザーの曖昧な思考を、**「実行可能な順序関係を持つアクションリスト」**に変換することです。

### 1. システムプロンプト（AIへの命令書）のドラフト案

AIに対して、以下のような役割と制約を与えます。

> **Role:** あなたは経験豊富なプロジェクトマネージャーです。
> **Task:** ユーザーの抽象的な目標を、具体的かつ「MECE（漏れなく重複なく）」なアクションに分解してください。
> **Rules:**
> 1. 各タスクは「〜する」という動詞形式で終わる具体的なアクションにすること。
> 2. タスク間に順序が必要な場合（例：予約の前に空き確認が必要など）、`depends_on`フィールドに先行タスクのIDを指定すること。
> 3. **「純粋な論理性」のみを重視**し、現時点では「いつやるか」や「場所」は考慮しない。
> 4. 出力は以下のJSON形式に厳密に従うこと。
>
```json
    [
        {
            "id": 1,
            "task": "タスク名",
            "description": "詳細説明",
            "depends_on": [],
            "logical_importance": 5,
            "estimated_weight": 3
        }
    ]
```

### 2. ロジックの根拠（なぜこの形か）

* **IDベースの依存関係 (`depends_on`):**
    単なるリストではなく、タスク同士を線でつなぐ（グラフ構造にする）ことで、ユーザーに「今、着手できるタスク（先行タスクが完了済みのもの）」だけを提示できるようになります。
* **説明文 (`description`) の保持:**
    タスク名だけでは「何をするか」を忘れる可能性があるため、分解時にAIが考えた意図を言語化しておきます。
* **「いつ」を排除する:**
    ここで「期限」を考えさせると、AIの推論リソースが分散して分解の精度が落ちます。「何が必要か」に100%集中させます。


### 3. 上記プロンプトの現状の問題点
- タスクの「同時並行性」:
    - 依存関係がないタスク（TOEICのID:3の単語など）が複数あります。これらは「いつでも並行して進めて良い」というフラグとしてシステム側で扱う必要があります。
- 粒度のバラツキ:
    - 「単語2000語の習得」と「テストの申し込み」が並列になっています。これらは「完了」の定義が異なる（毎日継続するもの vs 1回で終わるもの）ため、フェーズ分けやタグ付けが必要かもしれません。

## 2. Step B：コンテキストマッピング（Attribute API）への橋渡し

Step Aで作られた「論理的な骨組み」を受け取り、**ユーザーの現実（コンテキスト）**を肉付けします。

### 属性付与のロジック案（仮）

ここでは、AIに以下の情報をインプットとして与えます。
1.  **Step Aの出力リスト**
2.  **ユーザーの現在状況:**
    * 今日の日付（例：2026年4月19日）
    * プロジェクトの最終期限（例：5月末まで）
    * ユーザーの拠点（例：東京都足立区）
    * ユーザーの属性（例：平日は夜だけ自由、移動は徒歩・電車）

**【AIに判定させる項目とその根拠】**
* **推定労力 (1-5):** タスクの重さを数値化。後のフェーズで「疲れている時は軽いもの」を提案するために必須。
* **場所タグ:** 「どこでも可」「自宅」「外出」などを判定。
* **推奨期限の逆算:** 依存関係の最後にあるタスクを「最終期限」とし、そこから依存関係を遡って各タスクの「推奨着手日」を算出します。
    $$T_{deadline}(i) = T_{final} - \sum_{j \in successors(i)} Effort(j)$$
    *(※実際にはもっと柔軟なバッファを持たせますが、論理的にはこのように逆算します)*


### システムプロンプト (System)

> **Role:** あなたは高度なパーソナル・スケジューラーです。
> **Task:** Step Aで生成された「論理タスクリスト」に対し、ユーザーの現実的な「コンテキスト」を考慮して属性を付与してください。
> **Rules:**
> 1. **Location:** [自宅, 外出, 移動中, どこでも] から最適なものを選択。
> 2. **Energy:** 必要な集中度や精神的負荷を 1（低い）〜5（高い）で判定。
> 3. **Task_Type:** 「一度きりの完了(one-time)」か「継続的な習慣(habit/routine)」かを判定。
> 4. **Date_Calculation:** ユーザーが提示する「今日の日付」と「最終期限」を使い、依存関係を考慮して各タスクの `start_date` と `due_date` を設定してください。
> 5. **JSON形式のみを出力してください。**

## 3. 優先度計算のコンセプト：Dynamic Score

このTODOリストの肝は、**「固定の優先度（高・中・低）」ではなく、状況（コンテキスト）によってスコアがリアルタイムに変動する**点にあります。

### スコア算出の構成要素
以下の4つの「係数」を掛け合わせ、または足し合わせることで、最終的な `PriorityScore` を算出します。

| 要素 | 変数名 | 内容 |
| :--- | :--- | :--- |
| **重要度係数** | $I$ (Importance) | Step Aで決まった `logical_importance` |
| **緊急度係数** | $U$ (Urgency) | `due_date` と現在時刻の距離 |
| **コンテキスト合致度** | $C$ (Context) | 現在地とタスクの `location` の一致 |
| **エネルギー適正度** | $E$ (Energy) | 自分の現在の活量とタスクの `energy_cost` の相性 |

---

### アルゴリズム（数式）の案

$$PriorityScore = (I \times w_1 + U \times w_2) \times C \times E_{match}$$

#### 変数定義
* $I$ (Importance): Step A で設定された論理的重要度（1-5）
* $w_1, w_2$: 初期値は **1.0**。ユーザー設定により変動可能。
* $C$ (Context):
    * 現在地がタスクの `location` と一致：**1.0**
    * タスクが「どこでも」：**0.8**
    * 一致しない：**0.0** （リストから非表示）
* $E_{match}$ (Energy Match):
    * 自分の活量 $\geq$ タスクの `energy_cost`：**1.2**（ブースト）
    * 自分の活量 $<$ タスクの `energy_cost`：**0.2**（抑制）

#### 緊急度 ($U$) の算出
単なる「残り日数」ではなく、**「残りの工数に対して、どれだけ時間に余裕がないか」**を計算します。
* $U = \frac{\text{必要工数 (estimated\_weight)}}{\text{残り時間 (due\_date - now)}}$
* これにより、期限が近くてもすぐ終わるタスクより、期限は少し先でも重たいタスクのほうが上位に来るようになります。

#### このアルゴリズムの利点

この計算式を導入すると、以下のような挙動が自動で実現します。

1.  **朝（元気な時）**: 「TOEIC模試（高負荷・自宅）」が1位に来る。
2.  **仕事帰りの電車内**: 模試は消え、「単語学習（低負荷・移動中）」が1位に浮上する。
3.  **期限前日**: 多少疲れていても、緊急度 ($U$) が跳ね上がるため、そのタスクが最上位に強制浮上する。

| カラム名 | 型 | 説明 |
| :--- | :--- | :--- |
| `w1_importance` | Float | $w_1$ (初期値 1.0) |
| `w2_urgency` | Float | $w_2$ (初期値 1.0) |
| `base_location` | String | 自宅の座標や住所 |
| `current_energy` | Integer | ライフログから算出された現在の活量 (1-5) |

### 実装上のポイント（根拠）

* **`depends_on` の制御**:
    プログラム側では、`depends_on` に含まれる全てのタスク ID の `status` が `done` になるまで、そのタスクを「着手可能（候補）」として扱わない処理を入れます。
* **進捗率の動的計算**:
    プロジェクトの進捗は、そのプロジェクトに紐づくタスクの `estimated_weight` の合計に対し、完了済みの合計を割ることで算出します。
* **物理削除しない**:
    AIが「このタスクは不要ではないか？」と提案してユーザーが消したタスクは、`status` を `archived`（非表示）にするだけでデータは残します。これが後の「学習」の材料になります。

## 4. AIによる断捨離・調整提案


### 1. 提案発動のトリガー（いつ話しかけるか）

AIが介入を判断する基準値を**「キャパシティ・スコア」**として定義します。

* **負荷合計 ($L$)**: その日に予定されているタスクの $(Estimated\_Weight \times Energy\_Cost)$ の総和。
* **限界値 ($C_{max}$)**: ユーザーの過去の平均完了データ ＋ 昨夜の睡眠ログなどから算出。
* **判定**: $$L > C_{max} \times 1.2$$ （許容範囲を20%超えたら介入）

---

### 2. ターゲット選定ロジック（何を削るか）

タスクを以下の3つのカテゴリーに自動分類し、優先度の低いものから提案対象にします。

| カテゴリー | 選定基準 | AIのアクション |
| :--- | :--- | :--- |
| **A. 期限延長（Snooze）** | 重要度は高いが、今日である必要がなく、かつ負荷が高いもの。 | 「来週に回しましょう」と提案。 |
| **B. タスク分解（Divide）** | 負荷が高すぎて、着手ハードルが心理的障壁になっているもの。 | 「まず15分だけやりませんか？」と最小化を提案。 |
| **C. 削除・凍結（Discard）** | 重要度が低く、かつ作成から一定期間（例：2週間）未着手のもの。 | 「これ、本当に必要ですか？」と削除を提案。 |

---

### 3. 具体的な対話シナリオ（どう伝えるか）

ユーザーが設定した「トーン（性格）」に合わせて、提案のニュアンスを変えます。

#### シナリオ：タスクが多すぎてパンクしそうな時
> **状況**: 22:00の夜の振り返り。未完了タスクが5件残っている。

* **【共感モード】**
    > 「今日もお疲れ様でした。少し予定を詰め込みすぎちゃいましたね。この『参考書を買う』タスクは明日でも全然大丈夫です。今はスマホを置いて、ゆっくり休みませんか？」
* **【論理モード】**
    > 「現在の完了率は40%です。睡眠不足は翌日のパフォーマンスを15%低下させます。重要度2以下のタスク3件を自動的に来週へリスケジュールし、明日の朝の時間を確保することを推奨します。実行しますか？」
* **【コーチモード】**
    > 「全部やり遂げたい気持ちはわかります。でも、一流は『捨て方』も一流です。今は一番大事な『模試の復習』に集中するために、他の雑務は一旦忘れましょう。私が整理しておきますね！」

---

### 4. 「断る」提案のステップ

AIが相談を持ちかける際のフローを以下のように設計します。

1.  **現状認識の共有**: 「今日は予定より時間が足りなくなっていますね」
2.  **具体的な代替案の提示**: 「このタスクを土曜日に移動すると、今日1時間余裕が生まれます」
3.  **ユーザーの意思確認**: 「移動させてもよろしいですか？」
4.  **実行と学習**: ユーザーが「Yes」ならリスケし、「No」なら「なぜ今日やりたいのか（例：明日が締切だから）」を学習し、次回の予測精度を上げます。

### 5. ステイル（塩漬け）タスクへの切り込み

ここがこのアプリのユニークな点です。

* **放置タスクへの問いかけ**:
    「この『プログラミング学習』、2週間一度も触れていませんね。今のあなたには優先順位が低いのかもしれません。一旦『いつかやるリスト』に隠して、今の画面からは消しちゃいませんか？」
* **根拠**: 視界に入る「やっていないタスク」は、それだけで脳のメモリ（ウィルパワー）を消費します。AIがこれを**「隠す」提案をすることで、ユーザーの集中力を守ります。**

# Phase 3：技術スタックと開発アプローチ

個人開発における「開発スピード」「コスト（無料枠の活用）」「AI親和性」を最優先した構成。

## 1. フロントエンド / バックエンド
* **フレームワーク:** **Next.js (App Router)**
    * **選定理由:** Reactのスキルを活かしつつ、API Routes（バックエンド処理）を同一プロジェクト内で管理できるため。TypeScriptによる厳密な型定義でAIとのJSON連携を安定させる。
* **スタイリング:** **Tailwind CSS**
    * **選定理由:** ユーティリティファーストな設計により、最小限のコードで「シンプルかつ清潔感のある」UIを高速に構築できる。

## 2. データベース / 認証
* **プラットフォーム:** **Supabase**
    * **選定理由:** PostgreSQLベースのリレーショナルDB。今回の「プロジェクト＞タスク」の階層構造や依存関係のクエリに最適。認証機能（Googleログイン等）も標準搭載されており、開発工数を大幅に削減できる。
* **プラン:** Free Tier（無料枠）内で運用。

## 3. AI / API 連携
* **メインLLM:** **Google Gemini API (Gemini 3.0 Flash/Pro)**
    * **選定理由:** 無料枠が非常に大きく、個人開発に優しい。特に「タスク分解」などの高速なレスポンスが必要な処理にFlashモデルが適している。
* **SDK:** `Google Generative AI SDK for JavaScript`

## 4. 開発・デプロイ
* **エディタ:** **github Copilot**
    * **アプローチ:** 本設計書をプロンプトとして読み込ませ、「スモールステップ」で実装を進める。
* **ホスティング:** **Vercel**
    * **選定理由:** Next.jsとの親和性が最も高く、GitHubと連携した自動デプロイが無料で利用可能。

## 5. 開発ロードマップ（推奨）
1.  **MVP開発:** ログイン ＞ 手動でのプロジェクト/タスク作成 ＞ 優先度順での表示。
2.  **AI統合:** Step A（分解）とStep B（属性付与）のプロンプト実装。
3.  **ライフログ連携:** 手動での「活量」入力による動的な優先度変動の実装。
4.  **介入実装:** キャパシティ・スコアに基づく「AI相談画面」の実装。
